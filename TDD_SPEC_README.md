## 概述

### 什么是TDD规格说明书？

TDD（测试驱动开发）规格说明书是一种结构化的需求文档格式，专为AI辅助开发设计。它将传统的PRD（产品需求文档）转化为机器可解析、人类可阅读的双语文档。

### 核心优势

| 传统PRD | TDD规格说明书 |
|---------|--------------|
| 自然语言描述 | 结构化Gherkin场景 |
| 难以自动化测试 | 直接生成测试代码 |
| 开发与测试分离 | 需求即测试用例 |
| AI难以理解 | AI可直接解析执行 |

### 适用场景

- ✅ AI辅助代码生成
- ✅ 测试驱动开发（TDD）
- ✅ 自动化测试用例生成
- ✅ 敏捷迭代开发
- ✅ 跨团队协作（PM/开发/测试/AI）

## 使用流程

### 完整流程图

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   PM填写    │───>│  团队评审   │───>│ AI生成测试  │
│  第1-2章    │    │  第3-4章    │    │  & 骨架代码  │
└─────────────┘    └─────────────┘    └──────┬──────┘
                                             │
┌─────────────┐    ┌─────────────┐    ┌──────▼──────┐
│  迭代优化   │<───│   测试通过  │<───│  开发实现   │
│  更新规格   │    │   验收完成  │    │  填充逻辑   │
└─────────────┘    └─────────────┘    └─────────────┘
```

### 详细步骤

#### Step 1: 创建规格文档

复制模板到项目 `docs/` 目录：

```bash
cp templates/TDD_SPEC_TEMPLATE.md docs/TDD_SPEC.md
```

#### Step 2: 填写需求元信息（PM）

```yaml
# 编辑 docs/TDD_SPEC.md 第1章

需求ID: REQ-XXX
需求名称: [简短描述性名称]
优先级: [P0/P1/P2/P3]
迭代: [Sprint X]
负责人: [姓名]
状态: [草稿/评审中/开发中/测试中/已完成]
```

#### Step 3: 编写用户故事（PM）

```gherkin
# 编辑 docs/TDD_SPEC.md 第2章

作为 [角色]
我希望 [功能]
以便 [价值]
```

#### Step 4: 定义验收标准（PM + 开发 + 测试）

```gherkin
AC1: [场景描述]
  Given [前置条件]
  When [动作]
  Then [期望结果]
```

#### Step 5: 详细用例设计（开发 + 测试）

```gherkin
@priority:high @type:unit
功能: [功能描述]
  场景: [场景名]
    Given [条件]
    When [操作]
    Then [断言]
```

#### Step 6: AI生成代码

使用文档末尾的AI提示词模板，复制到AI对话中。

#### Step 7: 开发迭代

1. 运行生成的测试（应当失败 - 红）
2. 实现最小代码使测试通过（绿）
3. 重构优化（重构）

## 角色职责

### 产品经理 (PM)

| 关注章节 | 输出物 | 工具 |
|----------|--------|------|
| 第1章 需求元信息 | 需求ID、优先级、范围 | Markdown编辑器 |
| 第2章 用户故事 | 6个用户故事 | Gherkin语法 |
| 第2章 AC验收标准 | 每个故事的AC | Given-When-Then |

**交付标准：**
- [ ] 用户故事符合INVEST原则
- [ ] 每个AC可独立测试
- [ ] 边界条件明确

### 开发人员

| 关注章节 | 任务 | 输出物 |
|----------|------|--------|
| 第3章 功能规格 | 细化用例 | Gherkin场景 |
| 第4章 接口规格 | 设计API | YAML接口定义 |
| 第5章 数据规格 | 设计模型 | Entity/Repository |
| AI提示词模板 | 生成代码 | 测试+实现代码 |

**交付标准：**
- [ ] 所有AC有对应测试用例
- [ ] 接口定义完整（请求/响应/错误码）
- [ ] 单元测试覆盖率 > 80%

### 测试人员

| 关注章节 | 任务 | 输出物 |
|----------|------|--------|
| 第3章 用例清单 | 补充边界值 | 测试数据 |
| 第5章 测试数据 | 准备数据集 | SQL/JSON数据 |
| 第6章 错误处理 | 验证异常场景 | 错误测试用例 |
| 第8章 测试策略 | 设计集成/E2E测试 | 测试计划 |

**交付标准：**
- [ ] 边界值全覆盖
- [ ] 异常场景有测试
- [ ] 集成测试覆盖主流程

### AI助手

| 输入 | 处理 | 输出 |
|------|------|------|
| 第3-4章规格 | 解析Gherkin | JUnit5测试代码 |
| 接口定义 | 生成骨架 | Controller/Service |
| 数据模型 | 生成实体 | Entity/Repository |
| 错误码定义 | 生成异常 | ExceptionHandler |

## 快速开始

### 5分钟上手

#### 1. 复制模板

```bash
# 在项目根目录执行
cp templates/TDD_SPEC_TEMPLATE.md docs/TDD_SPEC.md
```

#### 2. 填写基础信息（5分钟）

```yaml
# docs/TDD_SPEC.md 第1章

需求ID: REQ-USER-001
需求名称: 用户注册登录模块
优先级: P0
迭代: Sprint 1
负责人: 张三
状态: 开发中
```

#### 3. 编写第一个用户故事（5分钟）

```gherkin
# docs/TDD_SPEC.md 第2章

### US-1: 用户手机号注册
作为 新用户
我希望 通过手机号注册账号
以便 使用AI辅导功能

**验收标准 (AC):**
```gherkin
AC1: 有效手机号注册
  Given 用户输入有效手机号"13800138000"
  And 该手机号未注册
  When 请求发送验证码
  Then 验证码发送成功
  And 60秒内不可重复发送
```
```

#### 4. 定义详细用例（10分钟）

```gherkin
# docs/TDD_SPEC.md 第3章

#### UC-01: 用户注册

```gherkin
@priority:high @type:unit
功能: 用户通过手机号注册

  场景: 有效手机号注册
    Given 用户输入手机号 "13800138000"
    And 该手机号未注册
    When 请求发送验证码
    Then 返回成功状态
    And 数据库创建验证码记录
```
```

#### 5. 使用AI生成代码

复制以下Prompt到AI对话：

```markdown
基于以下需求规格，请生成代码：

1. **生成单元测试代码**
   - 使用JUnit5 + Mockito
   - 测试UserService.register方法
   - 覆盖正常注册、手机号已注册、格式错误场景

2. **生成实现代码**
   - User实体类
   - UserRepository接口
   - UserService.register方法
   - 先写骨架，确保测试编译通过

需求规格：
[Paste UC-01 Gherkin内容]
```

### 分阶段Prompt策略

#### 阶段1：生成测试（红）

```markdown
请基于以下Gherkin场景生成JUnit5单元测试：

[粘贴用例]

要求：
- 使用Given-When-Then注释结构
- 使用Mockito模拟依赖
- 包含边界值测试
- 测试类名：XxxServiceTest
- 方法名：shouldXxxWhenYyy

输出格式：
1. 测试类完整代码
2. 必要的Mock配置说明
```

#### 阶段2：生成骨架（绿的第一步）

```markdown
请基于以下测试代码生成实现骨架：

[粘贴测试代码]

要求：
- 只生成类定义和方法签名
- 方法体抛出不实现异常
- 确保测试可以编译（但会失败）

输出：
1. 接口定义
2. 实现类骨架
```

#### 阶段3：填充实现（绿）

```markdown
请实现以下方法使测试通过：

[粘贴测试代码]
[粘贴骨架代码]

要求：
- 最小实现使测试通过
- 不引入过度设计
- 添加必要注释
```

#### 阶段4：重构优化

```markdown
请重构以下代码：

[粘贴当前代码]

要求：
- 应用设计模式（如需要）
- 提取重复代码
- 优化命名
- 保持测试通过
```

## 最佳实践

### 1. 规格编写原则

#### ✅ 好的AC示例

```gherkin
AC1: 验证码60秒冷却
  Given 用户手机号"13800138000"已发送验证码
  When 60秒内再次请求发送
  Then 返回错误码"CODE_SEND_TOO_FREQUENT"
  And 提示"请60秒后重试"
  And 不生成新验证码
```

#### ❌ 差的AC示例

```gherkin
AC1: 验证码有冷却时间
  用户不能频繁发送验证码
```

### 2. 用例粒度控制

| 层级 | 用例数量 | 场景示例 |
|------|----------|----------|
| 用户故事 | 3-5个AC | 注册成功、已注册、格式错误 |
| 功能用例 | 每个AC 1-3个场景 | 有效手机号、无效手机号、空手机号 |
| 单元测试 | 每个场景3-5个断言 | 验证返回值、验证数据库、验证缓存 |

### 3. 命名规范

```
需求ID:    REQ-{模块}-{序号}     例: REQ-USER-001
用例ID:    UC-{序号}             例: UC-01
测试ID:    UT-{序号}             例: UT-001
错误码:    {模块}_{描述}         例: USER_PHONE_EXISTS
```

### 4. 版本管理

```bash
# 规格变更时创建分支
git checkout -b docs/update-tdd-spec

# 修改后提交
git add docs/TDD_SPEC.md
git commit -m "docs: update TDD spec - add UC-05 for edge cases"

# 代码实现对应提交
git add src/
git commit -m "feat: implement user registration (REQ-USER-001)

- Add UserService.register with validation
- Add UserRepository.findByPhone
- Unit tests: UT-001 to UT-004

Related to UC-01 in TDD_SPEC.md"
```

### 5. 与CI/CD集成

```yaml
# .github/workflows/tdd-check.yml
name: TDD Spec Check

on:
  pull_request:
    paths:
      - 'docs/TDD_SPEC.md'
      - 'src/**'

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Check test coverage
        run: |
          # 确保所有AC都有对应测试
          ./scripts/verify-tdd-coverage.sh
      
      - name: Run tests
        run: mvn test
```

## 常见问题

### Q1: 规格文档需要多详细？

**A:** 遵循"刚好足够"原则：
- PM负责用户故事和AC（第1-2章）
- 开发负责详细用例和接口（第3-4章）
- 测试负责测试数据和异常场景（第5-6章）

### Q2: AI生成的代码质量如何？

**A:** AI是辅助工具，不是替代品：
- ✅ AI擅长：样板代码、测试框架、API骨架
- ❌ AI不擅长：业务逻辑、性能优化、安全敏感代码
- 💡 建议：AI生成 → 人工Review → 迭代优化

### Q3: 如何处理需求变更？

**A:** 规格即代码：
1. 修改 `docs/TDD_SPEC.md`
2. 同步更新测试代码
3. 同步更新实现代码
4. 提交时关联规格变更

### Q4:  legacy项目能使用吗？

**A:** 可以渐进式引入：
1. 为新功能使用TDD规格
2. 逐步为旧功能补规格
3. 重构时优先补测试

### Q5: 与现有PRD流程冲突吗？

**A:** 不冲突，是增强：
- 传统PRD → 高层业务描述
- TDD规格 → 可执行技术规格
- 两者可以并存，TDD规格是对PRD的技术细化
